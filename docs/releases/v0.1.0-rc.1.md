# Release v0.1.0-rc.1

## Summary
This is the first release candidate (RC) for KoalaBye. It includes the MVP feature set for organizations, campaigns, cookie-free feedback forms, anonymous submissions, analytics, and CSV/JSON exports.

**Note:** This is a release candidate intended for private, local, or staging use to verify deployment operations. It is not necessarily a polished public production launch.

## Who Should Use This RC
- Developers and operators looking to evaluate KoalaBye.
- Teams wishing to test a staging deployment before the stable v0.1.0 release.

## Install via Docker Compose
Follow the [Deployment guide](../DEPLOYMENT.md) using the `docker-compose.staging.example.yml` configuration. Make sure to generate a secure `KOALABYE_SECRET` and set the appropriate `KOALABYE_BASE_URL`.

## Upgrade Notes
If you have been testing from the main branch, ensure you run backups before updating. Migrations will run automatically.

## Backup Warning
Always test your backups! An untested backup is not a reliable safety net. Please see [BACKUP_RESTORE.md](../BACKUP_RESTORE.md) for details on backing up the SQLite database and environment variables.

## Security & Privacy Notes
- No email is required for user accounts.
- Legal pages (`/legal/privacy` and `/legal/imprint`) are placeholders and must be replaced before any public production deployment.
- Passwords are hashed and stored securely; however, there is no email/password reset functionality yet. 
- Analytics and exports contain no IP addresses, raw user agents, or raw install tokens.

## Known Limitations
- Retention is manual; no automated scheduler exists yet.
- Legal pages are placeholders and must be replaced before public production.
- No email/password reset functionality.
- No passkeys or OAuth support.
- No custom domains support.
- No automated backup scheduler.
- Docker build must pass in CI/release environment before tagging.

## Verification Checklist
Review the [RELEASE_CHECKLIST.md](../RELEASE_CHECKLIST.md) to perform end-to-end smoke testing on a staging environment before attempting any production deployment.

## Rollback Notes
Rollbacks to older database schemas are not officially supported due to forward-only migrations. To roll back, restore a previously taken SQLite backup and start the previous Docker image version.
